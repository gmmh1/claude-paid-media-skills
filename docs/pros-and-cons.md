# Advantages and Disadvantages of Open-Sourcing This Skill Library

An honest assessment, written for both the people using this library and anyone deciding whether to publish something similar. None of this is meant to talk you out of it — it's meant to tell you where to place your trust and where guardrails are already built in versus where you still need your own.

**Update (v1.1.0):** the first version of this document was written as a list of risks with advice attached — "here's what you should do about it." As of v1.1.0, several of those mitigations are no longer just advice; they're built directly into all 98 skills. The table below reflects that: each risk is marked **Mitigated in this repo**, **Partially mitigated**, or **Structurally inherent** (meaning no amount of editing Markdown can eliminate it — only disclose and manage it).

## Advantages

| # | Advantage | Why it matters |
|---|---|---|
| 1 | **Free, structured domain expertise** | 98 skills covering six ad platforms would take a solo operator or small agency months to compile from scratch. This collapses that to one command (`install.sh` / `install.ps1`, whole library or a single domain). |
| 2 | **Transparency** | Every rule and workflow is plain, readable Markdown — not a black-box SaaS algorithm. You can audit exactly what logic is being applied before you trust its output on a real client account. |
| 3 | **No vendor lock-in** | It's just files. Works with any Claude Code-compatible agent, no proprietary format, no subscription. You can strip it out as easily as you installed it. |
| 4 | **Extensibility under MIT** | Anyone can fork, modify, rebrand, or build a commercial product on top of this without asking permission. That's the point of the license. |
| 5 | **Consistency across the library** | Every skill follows the same structure (Purpose → Currency note → Core Capabilities → Rules → Verification Gate → Related Skills). Learn the pattern once, navigate all 98 skills the same way. |
| 6 | **Composability** | The master-router / specialist / cross-platform layering means you can install one platform, one domain, or the whole thing, and it degrades gracefully — a missing skill just means one fewer routing option, not a broken system. |
| 7 | **Business enablement, not just execution** | The `*-client-services` skills specifically exist to help someone turn this into an actual freelance or agency offering — pricing, proposals, scope of work — which is a different (and rarer) thing to open-source than pure execution capability. |
| 8 | **Community correction potential** | A single maintainer will miss platform changes. A community with PR access can catch and fix them faster — `CONTRIBUTING.md` sets the review bar so that correction happens without introducing new risk. |
| 9 | **Built-in humility** | Every skill states plainly, in its own text, what it doesn't know and where it can be wrong. That's unusual for either AI tooling or agency playbooks, both of which tend to oversell confidence. |

## Disadvantages and Risks

| # | Risk | Why it matters | Status | What this repo does about it |
|---|---|---|---|---|
| 1 | **Platform drift** | Google, Meta, LinkedIn, and TikTok change campaign types, character limits, and policies constantly. These are static files. | Partially mitigated | Every skill carries a **Currency & Scope note** with a "last reviewed" date, stating plainly that mechanical specifics are a snapshot, not guaranteed current, and must be checked against live platform docs before real use. This makes staleness *visible* rather than *hidden* — it doesn't stop drift, but it stops the false confidence drift causes. |
| 2 | **No official endorsement** | Not affiliated with Anthropic, Google, Meta, LinkedIn, TikTok, or Amazon. Platform names are used descriptively only. | Mitigated in this repo | Stated in the root README, `docs/README.md`, and now in every single skill's Currency & Scope note — not just in one place someone could miss. |
| 3 | **No built-in enforcement** | "Rules" sections are guidance for the agent, not a technical guardrail. Nothing stops a human from ignoring a flag. | Mitigated in this repo | Every skill now has a **Verification Gate** rule requiring explicit human sign-off, against current platform documentation, before its output is used to spend real money, make a compliance claim, or go to a client as final. This is still advisory (an LLM instruction, not a code-level lock) — see the honest limit below. |
| 4 | **Compliance guidance isn't legal advice** | Getting this wrong in a regulated category (housing, credit, employment) carries real legal exposure, not just a disapproved ad. | Mitigated in this repo | `google-ads-compliance` and `paid-media-safety-compliance` both explicitly state they are not a substitute for legal counsel and instruct escalation of ambiguous/high-stakes cases — this is now stated as a rule, not left implicit. |
| 5 | **Exposed business logic** | If your edge was "I know how to price a paid media retainer," that edge is now public. | Partially mitigated | All five `*-client-services` skills now state directly that they're a generic starting template, not a competitive edge, and that real differentiation has to come from execution and customization. This can't be un-published — the mitigation is honest framing, not secrecy. |
| 6 | **Fork fragmentation** | No single canonical "correct" version once people fork and customize. | Partially mitigated | `CHANGELOG.md` tracks library-wide changes; each skill's "last reviewed" date lets you tell whether a fork is ahead of, behind, or diverged from upstream. It doesn't stop forking (nor should it) — it makes drift traceable. |
| 7 | **Prompt-injection / trust surface** | Skill files are natural-language instructions loaded directly into an agent's context. A careless or malicious PR could smuggle in unwanted instructions. | Mitigated in this repo | `CONTRIBUTING.md` sets an explicit review bar for exactly this: read full diffs, check for instructions that *weaken* the Verification Gate or steer output toward an unnamed party's interest, don't merge on a plausible-sounding description alone. |
| 8 | **Maintenance burden** | 98 files is a lot to keep current. Without active maintainers, this goes stale fast. | Structurally inherent | `CHANGELOG.md` and per-skill review dates make staleness honestly visible, but they don't create maintainers. This risk is only fully addressed by an actual maintenance commitment, which no documentation can substitute for. |
| 9 | **Over-trust risk** | Polished Markdown can read as more authoritative than it is. | Mitigated in this repo | The Currency & Scope note explicitly flags mechanical specifics as unverified, and the Verification Gate rule exists specifically to interrupt the "this looks done, ship it" instinct before real spend happens. |

## The honest limit of a documentation-level fix

Be clear-eyed about what "mitigated in this repo" actually means: every mitigation above is still an *instruction to an LLM*, not a code-level enforcement mechanism. A Verification Gate rule makes an agent more likely to pause and ask for human sign-off — it does not, and cannot, technically block a spend action the way a real approval workflow with actual permissions would. Risks #1, #3, and #9 in particular are reduced, not eliminated: platforms will still change without this repo knowing, an agent can still be instructed around a rule, and confident-sounding Markdown will still occasionally be wrong. If you're wiring this into something that can actually spend money or publish claims without a human in the loop, put a real approval gate in your own system — don't rely on a skill file's text alone to be that gate.

## The short version

This library compresses a lot of scattered domain knowledge into one consistent, auditable, freely-usable format, and as of v1.1.0 it also actively discloses its own limits and requires a human-verification step before consequential use — rather than leaving that as an exercise for the reader. That's real, structural improvement over the first version. What it can't do is stop being a snapshot of fast-moving platforms, and it can't replace legal review in regulated categories or a real approval workflow anywhere real money moves automatically. Use it to move fast on the 80% that's stable (frameworks, structure, methodology) and verify the 20% that changes constantly (exact limits, exact policies, exact platform features) against the source — the Verification Gate in every skill exists to make sure that verification step actually happens instead of getting skipped under deadline pressure.
